## Essential Constants

**Laxity**: `INIT=0x1000`, `PLAY=0x10A1`, `INSTRUMENTS=0x1A6B`, `WAVE=0x1ACB`
**SF2 Driver 11**: `SEQ=0x0903`, `INST=0x0A03`, `WAVE=0x0B03`, `PULSE=0x0D03`, `FILTER=0x0F03`
**Control**: `END=0x7F`, `GATE_ON=0x7E`, `GATE_OFF=0x80`, `TRANSPOSE=0xA0`

## Known Limitations

| Source → Driver | Accuracy | Status |
|----------------|----------|--------|
| SF2-exported → Driver 11 | 100% | ✅ Perfect (including SidFactory_II/Laxity) |
| Native Laxity NP21 → Laxity driver | 99.93-100% | ✅ Production |
| Native Laxity NP21 → Driver 11 | 1-8% | ⚠️ Use Laxity driver instead |

**Critical**: "SidFactory_II/Laxity" ≠ native Laxity! Check player-id: "SidFactory" = use Driver 11, "Laxity_NewPlayer_V21" = use Laxity driver

## Version History

**v3.5.34** (2026-05-23): **Clean architectural-limit errors for high-load CONV_FAIL files.** Crosswords and Magic_Sound (both load=$F000) were failing with cryptic `struct.pack 'H' overflow` errors several frames deep in `sf2_header_generator.create_tables_block()`. The 16-bit address space is insufficient: at load=$F000 with binary 2-3KB, only ~700 bytes remain to $FFFF — not enough for the SF2 edit area (orderlists + sequences + F2/F3/F4/F5 tables + shadow buffer; minimum ~$800 bytes). Added a guard at the top of both `_inject_laxity_raw_np21` AND `_inject_player_raw_minimal`: when `sid_la + len(c64_data) + 0x800 > 0x10000`, raise a clean `ConversionError(stage='raw-NP21/minimal-embed inject (high-load)', reason='...')`. Echo_Beat had this style of clean error since v3.5.25 (low-load); now Crosswords and Magic_Sound get the same. **C2 corpus unchanged** (these files don't produce SF2s either way) but error messages are now diagnostic instead of a deep stack trace. Documents architectural constraints: SF2 edit-area Block 3 column addresses are 16-bit, so the layout can't represent binaries that fill ~all 64KB. Canonical regression byte-identical: Stinsen (1909), Patterns (1793), Edie_Ball (637), Alliance (1283); 1032 tests pass. Version sync → 3.5.34.

**v3.5.33** (2026-05-23): **Gate extended: wave-copy NOP + 200-frame window — Patterns recovered (last fixable audio residual).** Patterns.sid was the final audio residual after v3.5.32. SF2 had 1793 writes (matching SID count) but 11 minor divergences at frames 169, 185, 201, 249, 265, 281 — all `osc2_attack_decay: $7F` (SF2) vs `$00` (SID), every 16 frames. Ablation isolated the cause: the wave-copy routine at translator-end writes 32 bytes from the SF2 edit area to `np21_wave_data_addr` ($6903) and `np21_note_addr` ($6963) on every PLAY tick. For Patterns, both addresses overlap with the file's INSTRUMENT TABLE region — the wave-copy stomps an instrument's AD byte with the SF2-edit-area wave column's `$7F` (the SF2 packed-format end-of-sequence terminator) at runtime. Wave-copy is a Stage 7 Phase B.1 feature for F3 (wave) edit propagation; for files where it corrupts critical binary data, disabling it is the correct tradeoff: audio correct, F3 propagation lost. Extended `_run_post_build_audio_gate()` with progressive revert strategy: (1) trace SF2 vs SID; if matches, done; (2) try reverting ch_seq_ptr patches; (3) try NOP'ing wave-copy JSR; (4) try both. Each step is one zig64 trace (~30ms at 200 frames). Bumped frame window 64→200 to catch late divergences (Patterns' diverges at frame 169). Track wave-copy JSR location at translator emit time: `self._wave_copy_jsr_offs: list[int]`. **Patterns: 11-diff → byte-identical 1793 writes (gate NOPs wave-copy JSR after detecting ch_seq_ptr revert insufficient).** Edie_Ball, Joe_Gunn_Extras, SFd2, Alliance, Racer, Dark_Fun, Twone_Five, SFd1 all still PASS.
